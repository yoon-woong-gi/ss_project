# DRAM.OP — 기획/설계 문서

> 이 문서는 현재 저장소에 구현되어 있는 "DRAM READ/WRITE 동작 시각화 대시보드"를
> **코드가 하나도 없는 상태에서 새로 기획한다면 어떻게 설계했을지**를 역으로 정리한
> 기획서입니다. 실제 구현(`index.html`, `dram-dashboard.js`, `dram-dashboard.css`)을
> 분석해 그 안에 녹아 있는 설계 결정을 기획 문서 포맷으로 재구성했습니다.
> 신규 기능을 추가하거나 리뉴얼할 때 참고용 베이스라인으로 사용할 수 있습니다.

---

## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | DRAM.OP |
| 한 줄 정의 | DDR4 1T1C DRAM 셀 배열에서 READ / WRITE 명령이 어떻게 동작하는지 단계별로 시각화하는 인터랙티브 학습용 대시보드 |
| 형태 | 정적 웹앱 (빌드 도구 없음, 서버 없음, 로컬 파일/정적 호스팅으로 바로 구동) |
| 사용자 | 반도체/컴퓨터구조 전공 학생, DRAM 설계·검증 엔지니어, 사내 온보딩 교육 자료를 찾는 실무자 |
| 핵심 가치 | "말로 설명하면 이해하기 어려운 타이밍/전압 동작을 그림 + 애니메이션 + 숫자로 동시에 보여준다" |

## 2. 문제 정의 & 목표

- DRAM ACT → RD/WR → PRE 커맨드 시퀀스와 tRCD, tRAS, tRP, CL, CWL 같은 타이밍 파라미터는
  교과서 텍스트만으로는 직관적으로 와닿지 않는다.
- 워드라인 전압, 비트라인 charge sharing, 센스앰프 래치 시점처럼 "눈에 안 보이는" 아날로그
  동작을 시각적으로 보여줄 도구가 필요하다.
- **목표**
  1. READ/WRITE 두 시나리오를 각각 7단계 내외의 스텝으로 쪼개 재생/일시정지/앞뒤 이동이
     가능한 "커맨드 애니메이션 플레이어"를 만든다.
  2. 각 스텝마다 (a) 셀 배열 그림, (b) 워드라인/비트라인 전압 파형, (c) 커맨드 버스, (d) 타이밍
     테이블을 한 화면에서 동기화해서 보여준다.
  3. 전공 용어(tRCD, tRAS, CL, VccVt 등)에 마우스를 올리면 즉시 설명이 뜨도록 해 별도 검색을
     없앤다.
  4. 한국어/영어 전환, 라이트/다크 테마를 지원해 강의·발표 환경에서도 바로 쓸 수 있게 한다.
- **비목표(Out of scope)**: 실제 DRAM 컨트롤러 시뮬레이션(정확한 사이클 단위 RTL 재현), 사용자
  로그인/서버 저장, 다른 메모리(SRAM, NAND) 지원 — 필요 시 후속 과제로 분리.

## 3. 정보 구조(IA) & 화면 설계

원페이지(단일 `index.html`) 구조. 화면을 상단 헤더 / 중앙 스테이지 / 우측 사이드 패널 / 하단
트랜스포트(재생 컨트롤) 4개 영역으로 나눈다.

```
┌─────────────────────────────────────────────────────────────────┐
│ HEADER  브랜드 · [READ | WRITE] 탭 · 언어토글 · 테마토글 · 메타정보 │
├───────────────────────────────────────┬───────────────────────────┤
│                                       │ STEP 패널 (n/총)          │
│           STAGE (SVG 시각화)          │  - 현재 커맨드명, 설명     │
│   셀 배열 grid + 워드라인/비트라인/    ├───────────────────────────┤
│   센스앰프 파형 + DQ 버스              │ COMMAND BUS 패널           │
│                                       ├───────────────────────────┤
│           범례(legend)                │ TIMING 패널 (파라미터 표)  │
├───────────────────────────────────────┴───────────────────────────┤
│ FOOTER  RESET · PREV · PLAY/PAUSE · NEXT · 진행바 · 스텝 카운터   │
└─────────────────────────────────────────────────────────────────┘
```

- **탭(시나리오 전환)**: READ / WRITE 두 개. 탭 전환 시 스테이지 SVG, 스텝 패널, 타이밍 표,
  헤더 프로파일 텍스트가 모두 해당 시나리오 데이터로 리셋된다.
  (색상 토큰에 `--refresh`가 이미 정의돼 있어, REFRESH 시나리오를 세 번째 탭으로 추가하는 것을
  애초에 확장 포인트로 설계해 둔 상태 — 4번 향후 확장 참고)
- **스테이지**: `viewBox 900x650` SVG 한 장에 8행×16열 셀 그리드, 대상 셀(강조), 워드라인,
  비트라인 쌍(BL/BL̄), 센스앰프 박스, DQ 버스, 전압 파형(Vcc+Vt, Vcc, Vref, Gnd 기준선)을 모두
  좌표 상수로 배치. 반응형 컨테이너 안에서 `preserveAspectRatio`로 스케일링.
- **사이드 패널 3분할**: STEP(현재 단계 설명) / COMMAND BUS(발행된 커맨드 칩 목록) / TIMING
  (해당 시나리오의 tRCD·tRAS·tRP·CL 등 파라미터 표, 활성 파라미터 하이라이트).
- **트랜스포트**: RESET(0단계로) · PREV · PLAY(자동 재생, 인터벌 타이머) · NEXT · 진행률 바 ·
  "현재/전체" 스텝 카운터. 재생 중에는 PLAY 버튼이 PAUSE로 토글.
- **툴팁**: 본문·설명 텍스트 중 전공 용어를 정규식으로 감지해 밑줄 처리하고, hover/focus 시
  좌표 기반 플로팅 툴팁으로 용어 설명 노출 (키보드 접근성 위해 focusin/focusout도 처리).

## 4. 인터랙션 & 상태 설계

- **전역 상태 객체**(`S`)에 다음을 보관: 현재 시나리오, 현재 스텝 인덱스, 재생 타이머 핸들,
  현재 언어, 현재 테마.
- **상태 전이**
  - 시나리오 변경 → 스텝 인덱스 0으로 리셋, 진행 중인 재생 타이머 정지.
  - PLAY → `setInterval`로 일정 주기마다 NEXT 트리거, 마지막 스텝 도달 시 자동 정지.
  - NEXT/PREV → 스텝 인덱스 변경 → 렌더 함수 재호출(SVG 셀 상태, 파형, 커맨드 버스 칩,
    타이밍 하이라이트, 스텝 설명 텍스트 모두 재렌더).
  - 키보드 단축키: 방향키(PREV/NEXT), Space(PLAY/PAUSE), 탭 전환용 키 바인딩까지 고려
    (`keydown` 리스너 2곳: 전역 트랜스포트 제어 + 탭 접근성 이동).
- **접근성**: 탭에 `role="tablist"/"tab"`, 툴팁에 `role="tooltip"`, `aria-hidden` 토글,
  `prefers-reduced-motion` 매체쿼리로 애니메이션 최소화 옵션 지원.
- **반응형 브레이크포인트**: `1120px`(사이드 패널을 스테이지 아래로 재배치), `600px`(모바일
  단일 컬럼, 트랜스포트 버튼 축약) 2단계.

## 5. 콘텐츠 모델 (데이터 설계)

시각화 로직과 콘텐츠(교육 데이터)를 분리해서, **시나리오 = 데이터**로 선언하고 렌더러는
그 데이터를 소비만 하도록 설계한다.

```
scenarios = {
  read: {
    profile: { en, ko },              // 헤더에 표시할 프로파일 문자열(예: DDR4-3200 · CL22)
    timings: [ { name, valueCycles, valueNs, desc:{en,ko} }, ... ],
    steps: [
      {
        op, title:{en,ko}, desc:{en,ko},
        bus: ["ACT"] | ["ACT","RD"] | ...,   // 이 스텝에서 활성화된 커맨드 버스 표시
        cellState, waveformState, ...        // 렌더러가 SVG에 반영할 상태
      },
      ... (READ 7스텝: Idle → ACT → row 전개 → 센스앰프 래치 → RD → 데이터 버스 구동 → PRE)
    ]
  },
  write: { ...동일 구조, WR 관련 8스텝... },
  // refresh: (예약된 확장 슬롯, 3번째 색상 토큰까지 이미 준비됨)
}
```

- `TERM_DEFS`: 전공 용어 사전 (tRCD, tRAS, tRP, tRFC, tREFI, tREFW, CL, CWL, BL8, Vcc 등) —
  각 항목에 en/ko 설명 텍스트. 본문에서 이 키워드가 나오면 자동으로 툴팁 트리거로 치환.
- `UI`: 헤더/범례/패널 라벨 등 고정 UI 문자열의 en/ko 사전 — 언어 토글 시 DOM 텍스트를
  일괄 치환.
- 모든 텍스트가 `{en, ko}` 쌍으로 존재 → 국제화(i18n)를 "번역 시스템"이 아니라 "데이터에
  내장된 필드"로 단순하게 처리(오버엔지니어링 방지, 정적 사이트 규모에 맞는 선택).

## 6. 비주얼 디자인 시스템

- **테마**: CSS 커스텀 프로퍼티(`:root` 다크 기본값 + `[data-theme]`류 오버라이드로 라이트)로
  다크/라이트 2테마. 배경(`--void`), 경계선(`--rule`), 그림자(`--shadow`) 등 표면 토큰과
  READ(cyan) / WRITE(amber) / REFRESH(green) 시나리오별 강조색(`--read`, `--write`,
  `--refresh` + rgb 변형)을 분리해, 시나리오 전환 시 강조색만 스와핑.
- **타이포그래피**: 본문은 sans, 수치·커맨드·레지스터명 등 "계측기스러운" 요소는 전부 mono
  폰트 — 오실로스코프/로직 애널라이저 같은 계측 장비 룩앤필을 목표로 함.
- **톤**: 다크 배경 + 네온 계열 강조색 + 모노스페이스 수치 = 반도체 계측 장비/터미널 감성.
- **모션**: 셀 활성화·파형 스윕 등은 CSS transition/keyframe 기반, `prefers-reduced-motion`
  대응.

## 7. 기술 스택 & 아키텍처

| 레이어 | 선택 | 이유 |
|---|---|---|
| 마크업 | 순수 HTML (`index.html`) | 배포/실행에 빌드 스텝이 필요 없는 단일 정적 페이지 |
| 스타일 | 순수 CSS + CSS 변수 | 프레임워크 없이 테마 토큰 시스템만으로 다크/라이트 충분히 커버 |
| 로직/렌더링 | 순수 Vanilla JS, 단일 IIFE (`dram-dashboard.js`) | 상태-렌더 사이클이 단순(스텝 인덱스 → DOM/SVG 갱신)해 React 등 프레임워크 도입 비용이 정당화되지 않음 |
| 시각화 | 수기 좌표 기반 SVG (라이브러리 없음) | 셀 그리드·파형이 고정된 커스텀 다이어그램이라 D3 등 범용 차트 라이브러리보다 직접 제어가 유리 |
| 빌드/배포 | 없음 (정적 파일 그대로 서빙) | 프로젝트 규모상 번들러 도입은 과함. GitHub Pages/정적 호스팅에 바로 올릴 수 있어야 함 |

**아키텍처 원칙**
1. **데이터-렌더 분리**: `scenarios`/`TERM_DEFS`/`UI` 같은 콘텐츠 데이터와, 이를 DOM/SVG에
   반영하는 렌더 함수들을 분리. 콘텐츠(스텝 문구, 타이밍 값)만 고치면 로직을 건드릴 필요가
   없도록 한다.
2. **단일 진실 소스(state) → 순수 렌더**: 상태 객체 `S`를 갱신한 뒤 관련 렌더 함수들을 다시
   호출하는 단방향 흐름. 양방향 바인딩/프레임워크 없이도 예측 가능한 갱신을 유지.
3. **좌표 상수 분리**: 그리드/파형 좌표(`GRID_X`, `STRIDE_X`, `D_WL`, `G_VCC` 등)를 파일
   상단에 명명된 상수로 모아, SVG 생성 로직에서 매직 넘버를 없앤다.


  로드 가능하다는 장점이 더 큼).

## 9. 비기능 요구사항

- **성능**: 외부 런타임 의존성 0개, 총 에셋 100KB 내외로 최초 로드 즉시 인터랙티브.
- **접근성**: 키보드만으로 탭/재생 컨트롤/툴팁 전부 조작 가능해야 함. `prefers-reduced-motion`
  존중.
- **국제화**: 문자열을 하드코딩하지 않고 en/ko 딕셔너리에서 가져온다 — 세 번째 언어 추가 시
  각 문자열 객체에 키만 추가하면 되는 구조.
- **반응형**: 데스크톱(사이드 패널 분리) → 태블릿(스테이지 아래 재배치) → 모바일(단일 컬럼)
  3단 대응.
- **브라우저 호환**: 최신 evergreen 브라우저 기준(SVG, CSS 커스텀 프로퍼티, `prefers-*`
  미디어쿼리 지원 전제).

## 10. 개발 로드맵 (마일스톤 관점)

1. **M1 — 정적 셸**: 헤더/스테이지/사이드/트랜스포트 레이아웃, 테마 토큰, 반응형 그리드.
2. **M2 — READ 시나리오**: 셀 배열 SVG, READ 7스텝 데이터, 스텝 네비게이션(PREV/NEXT/RESET),
   타이밍 패널.
3. **M3 — 재생 컨트롤 & 파형**: PLAY 자동재생, 워드라인/비트라인/센스앰프 전압 파형 애니메이션,
   진행률 바.
4. **M4 — WRITE 시나리오**: WR 시나리오 데이터 8스텝, 탭 전환 로직 일반화(READ 전용 코드를
   시나리오 매개변수화).
5. **M5 — 용어 툴팁 시스템**: `TERM_DEFS` 사전, 본문 자동 하이라이트 + 플로팅 툴팁,
   키보드 포커스 대응.
6. **M6 — 국제화 & 테마**: en/ko 토글, 라이트/다크 토글, 로컬 스토리지에 사용자 선택 저장(선택).
7. **M7 — 접근성/반응형 다듬기**: ARIA 속성, reduced-motion, 두 브레이크포인트 QA.

## 11. QA 체크리스트 (기획 단계에서 미리 정의)

- [ ] READ/WRITE 각 시나리오에서 스텝을 끝까지 재생했을 때 PLAY가 자동으로 멈추는가
- [ ] 언어 토글 시 헤더/패널/툴팁/타이밍 표까지 전부 en↔ko 동기화되는가
- [ ] 다크↔라이트 전환 시 시나리오 강조색(cyan/amber) 대비가 WCAG AA를 만족하는가
- [ ] 키보드만으로 탭 전환 → 스텝 재생 → 용어 툴팁 확인까지 전체 플로우가 가능한가
- [ ] 1120px, 600px 브레이크포인트에서 레이아웃 깨짐이 없는가
- [ ] `prefers-reduced-motion: reduce` 환경에서 애니메이션이 억제되는가

## 12. 향후 확장 아이디어

- **REFRESH 시나리오 추가**: 색상 토큰(`--refresh`)이 이미 준비돼 있으므로, 세 번째 탭으로
  REF 커맨드/tRFC/tREFI 흐름을 시각화하는 것이 가장 자연스러운 다음 확장.
- **타이밍 프로파일 선택**: 현재 DDR4-3200/CL22 고정 → DDR4-2400/2666/3200 등 프로파일
  드롭다운으로 확장해 JEDEC bin별 수치 비교.
- **뱅크/랭크 개념 도입**: 현재 단일 뱅크·단일 셀 배열 → 멀티뱅크 인터리빙 예제로 확장.
- **URL 상태 공유**: 현재 시나리오/스텝을 쿼리 파라미터에 반영해 특정 단계로 딥링크 공유.
- **콘텐츠 파일 분리**: `scenarios`/`TERM_DEFS` 데이터를 별도 JSON/JS 모듈로 분리해 비개발자도
  콘텐츠(설명 문구, 타이밍 값)를 리뷰·수정하기 쉽게.

## 13. 요약

이 프로젝트는 **"콘텐츠(교육 데이터) ↔ 렌더러(SVG/DOM) ↔ 상태(현재 스텝)"** 세 축을
철저히 분리한 순수 정적 웹앱으로 기획·구현되었다. 프레임워크나 빌드 도구 없이도 다국어,
테마, 스텝 재생, 용어 툴팁까지 갖춘 것은 "규모에 맞는 최소 기술 스택" 원칙을 따른 결과이며,
새 시나리오(REFRESH)나 새 프로파일을 추가할 때도 렌더러를 건드리지 않고 `scenarios` 데이터만
추가하면 되도록 설계된 것이 핵심 확장 포인트다.
